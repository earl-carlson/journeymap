Docker IA Map

May 2026
Foundation

Principles

01
Everything has a home.
Every feature, setting, and workflow has one primary home. If you're building something new and asking "where does this go," there is a clear answer. No squatting in the wrong space because it's convenient.
02
Each space is made for someone.
A developer working only on agentic tools only needs the Agentic Platform. An IT admin only needs Admin. The spaces are distinct enough that you know which one is yours without having to search. If someone has to look in three places to find their thing, the IA is wrong.
03
The system is modular, not duplicated.
Any surface can expose a contextual window into another — a billing widget, a doc panel, a usage summary. That window is a peek, not a copy. The full experience lives in one place. The widget is a convenience, not a second home.

Kill list

Things that should not exist or should be radically cut before being rebuilt.

Notification systems as currently implemented.
No new notification surface gets built until there is a clear answer to: what action does this enable that the user couldn't take without it?

Known gaps

Specific product and UX problems that are known and need fixing. Not IA decisions — things to fix.

Profiles and accounts.
Currently fragmented — profile settings are split across Hub, app.docker.com, and Admin with no single coherent home. A user managing their Docker identity visits multiple surfaces to do one job. Needs a defined primary home before any new profile features are built.
Add known UX and product gaps here as they're identified.

Status

Open questions

01
Gordon placement active
Gordon is owned by the Agents group but lives in Docker Desktop (Container Platform). The ia-map has it under Container Platform. The org has it under Agents. This tension needs a resolution — does Gordon have a presence in both, or does it move fully to Agentic Platform?
02
Coding Agent Sandboxes vs. Cloud Sandboxes active
Coding Agent Sandboxes (local, Dash Runtime group) is under Container Platform here. Cloud Sandboxes (Cloud Group) is under Agentic Platform. This split maps to the org but may be confusing to users who just think of "sandboxes." Worth a shared entry point or at least consistent naming.
03
Admin: one workflow or three? active
Admin serves procurement (billing), IT admins (provisioning), and security leads (governance) with pretty different goals. The navigation inside Admin probably needs three distinct entry points, not one flat list.
04
MCP placement active
MCP is owned by AI Tools & Security (Agents group) and listed under Agentic Platform here. But MCP Toolkit in Desktop has historically lived in Container Platform. Primary home: Agentic Platform. Secondary surface: Desktop. Needs to be explicit in both places.
05
Docs panels ownership parked
A platform team owns the component. Docs team owns the content. Which team decides where panels appear and what content they surface? Needs an owner before this scales.
06
Hub org ownership parked
Hub (Registry + Marketplace) is owned by the Container Platform group but kept as a standalone area in this map. Revisit once the product surface boundaries are clearer.
07
CLI placement parked
Listed as secondary under both Container Platform and Agentic Platform. Is that the right model or does it need a clearer primary home?
08
Governance Platform as infrastructure blocked
The Code Red doc is explicit that Governance is the substrate, not a sibling product. The ia-map reflects this in the Admin section but the full implications — what "plugging in" means for each product, what's hosted on app.docker.com — are still being defined by Steven/Brian.
09
DVP (Docker Verified Publisher) insights — missing journey parked
The product map PDF covers a distinct DVP surface: verified publishers get access to pull data on their images, separate from standard Hub repository management. This is a different role (ISV / commercial publisher) and a different job than the Hub Publisher journey here. Not yet represented in journeys.md or the IA map. Worth a dedicated journey if DVP is a meaningful segment.
10
Churn & contraction — missing from Billing journey parked
The PDF covers downgrade-to-free and seat removal as explicit flows (scheduled at end of billing cycle, no mid-tier downgrade supported). The Billing journey here covers subscribe, PAYG, seat addition, and usage review — but not the offboarding/contraction path. Low priority but worth adding when the Billing journey gets a deeper pass.

Journey 01
Local AI inference
Docker Model Runner
Persona: Developer
Persona — Alex
Role Developer building or integrating AI-powered features into their product
Experience Comfortable with Docker and CLI. Has used LLM APIs but hasn't run models locally before.
Goals Prototype AI features quickly, avoid cloud inference costs, keep data off third-party servers, avoid a new toolchain.
Frustrations Local model setup is fragmented — llama.cpp, Ollama, Hugging Face all separate from the container workflow. No standard way to version models alongside the stack.
Awareness
"I need a better way to run models locally"
Cloud costs are adding up, setup is messy, every model has a different download process. A Docker blog post or colleague mention lands. The pitch — run models like containers — resonates immediately because Docker is already central to their workflow.
Consideration
"Can this actually fit into how I work?"
Alex reads the docs and sees DMR uses the OpenAI-compatible API — no rewriting app code. Models are distributed as OCI artifacts on Docker Hub, the same registry they already use. The mental leap is small.
Activation
"Let me try it"
Updates Docker Desktop, enables Model Runner in settings, runs a model with a single docker model pull command. Hits the local API endpoint and gets a response in seconds using GPU acceleration. The moment it just works without separate installs is the aha moment.
Engagement
"I'm building with this"
Starts running models as part of the local dev loop, testing prompts, iterating on system instructions, integrating responses into the app. Uses Docker Compose to wire up the model alongside other services. Starts treating models like any other dependency.
Expansion
"My team should use this too"
Shares the workflow with the team. Platform engineer sees the value of OCI-based model distribution for standardizing versions across environments. Team starts pulling models from Hub as part of CI, potentially publishing their own fine-tuned model as an OCI artifact.
Journey 02
Contextual AI assistance
Gordon
Persona: Developer
Persona
Role Developer using Docker Desktop daily across containerized apps, Dockerfiles, Compose, and CI pipelines
Experience Competent with Docker but not an expert. Frequently hits walls — cryptic build errors, containers that fail to start. Not enough time to optimize images.
Goals Unblock quickly without leaving the current workflow. Get better at Docker over time without reading the entire manual.
Frustrations Context switching is expensive. Generic AI chatbots give plausible but subtly wrong Docker answers because they lack context about the actual environment.
Awareness
"Something broke and I need help"
Container fails to start. Notices Gordon right next to the container actions in Desktop, or a colleague mentions asking Gordon to optimize their Dockerfile. The entry point is frictionless — it's just there.
Consideration
"Is this actually useful or just another chatbot?"
Skepticism is high after being burned by generic AI answers. Notices Gordon has access to their actual environment — running containers, logs, local files, Docker Scout findings. That context-awareness is the differentiator.
Activation
"Let me try it on this error"
Opens the Ask Gordon panel on a failed container. Gordon reads the logs, identifies the issue, and suggests a fix with an explanation. No copy-pasting — the agent offers a solution and runs the commands. The answer is specific to their container. That first successful interaction is the aha moment.
Engagement
"I use this daily now"
Starts asking Gordon proactively: "Rate my Dockerfile," "How do I run MongoDB?", "What does this Scout policy violation mean?" Uses it in both Desktop and the CLI depending on where they're working. Gordon becomes the first stop instead of the browser.
Expansion
"The team should have this"
Team lead notices cleaner Dockerfiles and faster debugging. Enables Gordon for the team. As AI Governance is adopted, the platform engineer configures what Gordon can access and execute on behalf of developers, making it safe to roll out broadly.
Journey 07
Local to Kubernetes
Compose GUI
Persona: Developer · DevOps Engineer
Persona
Role Developer moving from local Compose development toward Kubernetes, or DevOps engineer validating a deployment path
Goals Inspect a running Compose stack visually, then convert and deploy to Kubernetes without leaving Desktop.

Journey 08
Build troubleshooting & optimization
Build UI in Desktop
Persona: Developer · DevOps Engineer
Persona
Role Developer or DevOps engineer debugging a failing build or trying to understand performance regressions
Goals Understand what happened in a build, fix it fast, and make informed decisions about performance without reading raw logs.
Stage 1
Debugging: Build fails — find out why
Triggers a build → it fails → opens Build UI → reads the build trace → identifies the failing layer → fixes the Dockerfile → rebuilds. Gordon is available inline to explain errors without leaving Desktop.
Stage 2
Performance: Compare builds to improve CI times
Focused on improving CI build times → uses Build UI to compare different builds side by side → understands performance impacts of Dockerfile changes → makes informed decisions before pushing to CI.
Journey 09
Accelerated CI builds
Docker Build Cloud
Persona: DevOps Engineer
Persona
Role DevOps engineer responsible for CI pipeline speed, reliability, and cost
Goals Radically reduce CI build times with shared cache. Understand build performance across the org. Debug complex pipelines without guessing.
Stage 1
Speed up CI: Shared cache, faster builds
Connects Build Cloud to the CI pipeline → shared cache kicks in across builds → build times drop significantly. Team stops waiting on cold builds.
Stage 2
Analytics: Understand build performance across the org
Reviews build performance across a time period, user group, or organization → identifies slow repositories and cache miss patterns → makes informed decisions about Dockerfile structure and CI configuration.
Stage 3
Pipeline debugging: Debug a complex CI pipeline
A multi-build CI pipeline is failing intermittently → accesses build metadata and troubleshooting analytics across the entire pipeline → identifies the root cause without manually correlating logs across jobs.
Journey 10
Cloud engine switching
Docker Offload
Persona: Developer · DevOps Engineer
Persona
Role Developer on constrained hardware, or DevOps engineer managing remote workers who need a consistent Docker environment
Goals Run Docker workloads without being limited by local hardware. Keep the local experience unchanged while compute runs in the cloud.
Stage 1
Individual dev: Offload from underpowered hardware
Developer on an older machine or VDI instance selects a cloud engine in Desktop → all Docker workloads run remotely → local experience is unchanged. No new toolchain, no config changes to the project.
Stage 2
Team rollout: Enable offload for a remote team
DevOps engineer enables offload for a team of remote workers on secure VDI instances → developers get consistent build and run performance regardless of local hardware → no disruption to existing workflows.
Stage 3
AI workloads: GPU-focused cloud engine for AI
Developer working on AI features selects a GPU-focused cloud engine → runs all AI workloads through Offload regardless of local GPU support → no separate cloud account or infra to manage.
Journey 11
Agentic isolation & governance
Docker Sandboxes
Persona: Developer · DevOps Engineer · IT Admin
Persona
Role Developer running agents locally, DevOps engineer managing agentic CI, or IT Admin setting org-level guardrails
Goals Run agents safely without risking the local filesystem or network. Enable parallel agent threads. Give admins control over what agents can do.
Stage 1
Local isolation: Run an agent without risking local state
Developer isolates a coding agent in a local sandbox → agent runs in YOLO mode to accomplish a task without risking the local filesystem or making rogue network calls.
Stage 2
Parallel threads: Multiple agents on the same codebase
Engineer runs multiple agent threads on the same working directory using --branch → each thread is isolated → no local file conflicts while sharing the same codebase.
Stage 3
Cloud sandboxes: Agentic isolation across CI and cloud
DevOps engineer uses sbx --cloud to enable agentic isolation across local, CI, and public views → consistent governance model regardless of where the agent runs.
Stage 4
Org governance: Set guardrails for the whole org
IT Admin configures agentic controls in Admin Console → decides which levels of sandbox enforcement are required or optional → development team uses agents safely and predictably within those bounds.
Journey 12
Secure base images
Docker Hardened Images
Persona: Platform Engineer · Security Lead · Developer
Persona
Role Platform or security engineer reducing vulnerability surface across the org, or developer pulling a hardened base image for a regulated environment
Goals Swap in hardened base images, mirror them to an internal registry, and apply org-specific customizations without disrupting developer workflows.
Stage 1
Catalog: Browse and select hardened images
Security engineer audits services across the org → browses the DHI catalog → identifies hardened equivalents for current base images → selects images with the right compliance profile (CIS, FIPS, STIG).
Stage 2
Mirroring: Mirror to internal registry
Platform engineer configures mirroring → DHI images are copied to the org's private registry → developers pull from the internal mirror without touching the public catalog directly. Keeps the supply chain clean.
Stage 3
Customizations: Apply org-specific changes
Platform engineer creates a customization — adds content or changes configuration on top of a DHI image via the UI → customization triggers a rebuild → the resulting image is available in the org's mirror with the changes baked in.
Journey 03
Autonomous development
Agentic Platform
Persona: Tech Lead
Persona
Role Tech lead managing multiple repos while also writing new features, handling PR reviews, and keeping docs in sync
Experience Comfortable with GitHub, Docker, and AI coding tools. Has used Copilot and Claude Code but finds them too babysitter-intensive for end-to-end work.
Goals Offload repetitive, well-defined coding work to autonomous agents without losing visibility. BYOK for billing control. Agents that behave like specialists, not chat tools.
Frustrations Existing AI tools require staying in the loop for every step. No way to run agents on a schedule or tie them to GitHub workflow without complex infra. Too many tools managed separately.
Awareness
"I need agents that actually do the work, not just suggest it"
Hears about the platform through Docker's blog, a colleague, or a conference talk. The promise — agents that take action on a real codebase, securely, and surface results as PRs — is different from chat-based AI.
Consideration
"Can I trust this with my repo?"
Signs in with Docker ID — no new account. Onboarding orients around: project ties together a repository, a team of agents, and a session history. Connects a GitHub repo with clearly scoped permissions: read, create PRs, read issues.
Activation
"Let me see what it can do"
Describes a task in plain language. Selects an agent. Agent runs in a secure cloud sandbox and produces a PR. The moment she reviews that PR and finds it credible is the aha moment.
Engagement
"I'm customizing this for how we work"
Creates a custom reviewer agent with specific instructions for her codebase, enables the GitHub capability, connects Slack for status updates. Uses the BYOK flow to connect her Anthropic API key, keeping billing and usage under her control.
Expansion
"This is now part of our routine"
Sets up a daily schedule — reviewer agent on all open PRs at 9am, weekly documentalist run for the CHANGELOG. Engineering manager sees the session history and artifacts, rolls the platform out to the broader team.
Retention
"Agents as teammates"
The platform handles PR reviews, issue triaging, test writing, and doc maintenance. Monitors sessions the way she'd monitor CI — checks in when something fails or needs approval, and mostly trusts the agents to handle the rest.

Journey 14
Subscription & spend management
Billing
Persona: Procurement · IT Admin
Persona
Role Procurement owner managing Docker spend, or IT Admin handling license quantities as the team grows
Goals Subscribe to the right plan, control PAYG spend, keep seat counts accurate, and understand what the org is paying for.

Subscribe: Choose a plan and add-ons
Procurement subscribes to Pro, Team, or Business → adds Gordon and/or DHI as add-ons if needed → selects annual or monthly billing cycle → completes checkout.

Usage review: Understand what the org is spending
Reviews usage dashboards at end of month — build minutes, Offload sessions, agent runs — against committed spend. Downloads invoices. Adjusts plan or quantities for the next cycle.
