Docker IA Map
May 2026
Foundation
Principles
Everything has a home.

Every feature, setting, and workflow has one primary home. If you're building something new and asking "where does this go," there is a clear answer. No squatting in the wrong space because it's convenient.

Each space is made for someone.

A developer working only on agentic tools only needs the Agentic Platform. An IT admin only needs Admin. The spaces are distinct enough that you know which one is yours without having to search. If someone has to look in three places to find their thing, the IA is wrong.

The system is modular, not duplicated.

Any surface can expose a contextual window into another — a billing widget, a doc panel, a usage summary. That window is a peek, not a copy. The full experience lives in one place. The widget is a convenience, not a second home.

Kill list

Things that should not exist or should be radically cut before being rebuilt.

Notification systems as currently implemented. No new notification surface gets built until there is a clear answer to: what action does this enable that the user couldn't take without it?

Known gaps

Specific product and UX problems that are known and need fixing. Not IA decisions — things to fix.

Profiles and accounts. Currently fragmented — profile settings are split across Hub, app.docker.com, and Admin with no single coherent home. A user managing their Docker identity visits multiple surfaces to do one job. Needs a defined primary home before any new profile features are built.

Add more

Add known UX and product gaps here as they're identified.

Key workflows
Debugging a failed build

Engineer triggers a build → it fails → opens Build UI → reads the build trace → identifies the failing layer → fixes the Dockerfile → rebuilds. Gordon is available inline to explain errors or suggest fixes without leaving Desktop.

Local model inference — Model Runner

Developer enables Model Runner in Desktop settings → pulls a model with docker model pull → hits the local OpenAI-compatible API endpoint → integrates into their app using the same API client they already use for cloud inference. Model is distributed as an OCI artifact from Hub.

Compose to Kubernetes

Developer runs a multi-container app locally with docker compose up → inspects the running stack in the Compose File Viewer → navigates to "Convert and Deploy to Kubernetes" → Compose Bridge generates Kubernetes manifests → deploys to the local Desktop Kubernetes cluster → validates pod and service behavior.

Offload

Engineer on underpowered hardware selects a cloud engine in Desktop → all Docker workloads run remotely → local experience is unchanged. DevOps manager enables offload for a team of remote workers on VDI instances.

Hardened Images

Security or platform engineer browses the DHI catalog → selects a base image → configures mirroring to internal registry → applies org-specific customizations → developers pull from internal mirror without touching the public catalog directly.

Local coding agent sandbox

Developer isolates a coding agent in a local sandbox → agent runs without risk to local filesystem or network. Engineer runs multiple agent threads on the same working directory using --branch for isolation without file conflicts.

Status
Open questions
01
Gordon placement
Active

Gordon is owned by the Agents group but lives in Docker Desktop (Container Platform). The ia-map has it under Container Platform. The org has it under Agents. This tension needs a resolution — does Gordon have a presence in both, or does it move fully to Agentic Platform?

02
Coding Agent Sandboxes vs. Cloud Sandboxes
Active

Coding Agent Sandboxes (local, Dash Runtime group) is under Container Platform here. Cloud Sandboxes (Cloud Group) is under Agentic Platform. This split maps to the org but may be confusing to users who just think of "sandboxes." Worth a shared entry point or at least consistent naming.

03
Admin: one workflow or three?
Active

Admin serves procurement (billing), IT admins (provisioning), and security leads (governance) with pretty different goals. The navigation inside Admin probably needs three distinct entry points, not one flat list.

04
MCP placement
Active

MCP is owned by AI Tools & Security (Agents group) and listed under Agentic Platform here. But MCP Toolkit in Desktop has historically lived in Container Platform. Primary home: Agentic Platform. Secondary surface: Desktop. Needs to be explicit in both places.

05
Docs panels ownership
Parked

A platform team owns the component. Docs team owns the content. Which team decides where panels appear and what content they surface? Needs an owner before this scales.

06
Hub org ownership
Parked

Hub (Registry + Marketplace) is owned by the Container Platform group but kept as a standalone area in this map. Revisit once the product surface boundaries are clearer.

07
CLI placement
Parked

Listed as secondary under both Container Platform and Agentic Platform. Is that the right model or does it need a clearer primary home?

08
Governance Platform as infrastructure
Blocked

The Code Red doc is explicit that Governance is the substrate, not a sibling product. The ia-map reflects this in the Admin section but the full implications — what "plugging in" means for each product, what's hosted on app.docker.com — are still being defined by Steven/Brian.

09
DVP (Docker Verified Publisher) insights — missing journey
Parked

The product map PDF covers a distinct DVP surface: verified publishers get access to pull data on their images, separate from standard Hub repository management. This is a different role (ISV / commercial publisher) and a different job than the Hub Publisher journey here. Not yet represented in journeys.md or the IA map. Worth a dedicated journey if DVP is a meaningful segment.

10
Churn & contraction — missing from Billing journey
Parked

The PDF covers downgrade-to-free and seat removal as explicit flows (scheduled at end of billing cycle, no mid-tier downgrade supported). The Billing journey here covers subscribe, PAYG, seat addition, and usage review — but not the offboarding/contraction path. Low priority but worth adding when the Billing journey gets a deeper pass.

Journey 01
Local AI inference
Docker Model Runner
Developer
Role
Developer building or integrating AI-powered features into their product
Experience
Comfortable with Docker and CLI. Has used LLM APIs but hasn't run models locally before
Goals
Prototype AI features quickly, avoid cloud inference costs, keep data off third-party servers, avoid a new toolchain
Frustrations
Local model setup is fragmented — llama.cpp, Ollama, Hugging Face all separate from the container workflow. No standard way to version models alongside the stack
Awareness
"I need a better way to run models locally"

Cloud costs are adding up, setup is messy, every model has a different download process. A Docker blog post or colleague mention lands. The pitch — run models like containers — resonates immediately because Docker is already central to their workflow.

Consideration
"Can this actually fit into how I work?"

Alex reads the docs and sees DMR uses the OpenAI-compatible API — no rewriting app code. Models are distributed as OCI artifacts on Docker Hub, the same registry they already use. The mental leap is small.

Activation
"Let me try it"

Updates Docker Desktop, enables Model Runner in settings, runs a model with a single docker model pull command. Hits the local API endpoint and gets a response in seconds using GPU acceleration. The moment it just works without separate installs is the aha moment.

Engagement
"I'm building with this"

Starts running models as part of the local dev loop, testing prompts, iterating on system instructions, integrating responses into the app. Uses Docker Compose to wire up the model alongside other services. Starts treating models like any other dependency.

Expansion
"My team should use this too"

Shares the workflow with the team. Platform engineer sees the value of OCI-based model distribution for standardizing versions across environments. Team starts pulling models from Hub as part of CI, potentially publishing their own fine-tuned model as an OCI artifact.

Builds an image locally → tags it → pushes to Hub → sets visibility (public or private) → writes a repository overview so others know how to use it. Team pulls it as a dependency.

Repository management
Manage tags, webhooks, and access

Manages tags across multiple image variants → sets up webhooks for downstream automation → controls repository access for the org namespace. Monitors pull counts and storage usage via Hub insights.

Journey 14
Subscription & spend management
Billing
Procurement · IT Admin
Role
Procurement owner managing Docker spend, or IT Admin handling license quantities as the team grows
Goals
Subscribe to the right plan, control PAYG spend, keep seat counts accurate, and understand what the org is paying for
Subscribe
Choose a plan and add-ons

Procurement subscribes to Pro, Team, or Business → adds Gordon and/or DHI as add-ons if needed → selects annual or monthly billing cycle → completes checkout.

PAYG
Enable and control on-demand spend

Enables PAYG for Build Cloud or Offload → sets a spend limit → monitors usage against committed spend → adjusts on-demand settings if the team is burning through budget faster than expected.

Seat management
Add or remove seats as the team changes

Team grows → adds seats mid-cycle → prorated charge applied. Team contracts → schedules seat removal at end of billing cycle. Downgrade to free is scheduled, not immediate.

Usage review
Understand what the org is spending

Reviews usage dashboards at end of month — build minutes, Offload sessions, agent runs — against committed spend. Downloads invoices. Adjusts plan or quantities for the next cycle.
